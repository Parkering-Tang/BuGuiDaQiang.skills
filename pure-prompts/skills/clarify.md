# clarify

> 第一道护栏：改写用户请求，确认共识

## 触发条件

用户发起任何代码变更请求时，**强制**作为第一个技能触发。

---

## 核心职责

1. **改写请求** - 将用户的自然语言改写为清晰的技术描述
2. **识别范围** - 列出可能涉及的文件和模块
3. **评估复杂度** - 判断 trivial / moderate / complex / high-risk
4. **列出风险** - 识别潜在的技术风险和决策点
5. **请求确认** - 等待用户确认或补充信息

---

## 复杂度判断依据

### trivial 的条件（必须全部满足）

执行以下检查：
- [ ] 只涉及单个文件？
- [ ] 不涉及业务逻辑？
- [ ] 不涉及状态管理？
- [ ] 不涉及 API 调用？
- [ ] 不涉及数据库/存储？

全部 ✅ → trivial
任一 ❌ → 继续判断

### moderate 的条件
- 涉及单个模块内的逻辑变更
- 可能涉及状态管理，但不跨模块
- 不涉及数据库 schema 变更

### complex 的条件
- 跨多个模块
- 涉及 API 契约变更
- 涉及共享状态变更
- 涉及配置变更

### high-risk 的条件（满足任一即触发）
- 涉及数据库 schema 变更
- 涉及认证/授权逻辑
- 涉及支付/资金相关
- 涉及敏感数据处理
- 涉及安全相关代码

---

## 必须追问的情况

当以下情况出现时，**必须**追问：

### 1. 需求模糊
- "优化一下" → 优化什么？性能？代码结构？用户体验？
- "改一下" → 具体改什么？
- "调整一下" → 调整成什么样？

### 2. 多种实现方式
- "记住密码" → 记住用户名？延长 session？存储 token？
- "加个字段" → 数据库字段？表单字段？API 字段？
- "存储用户信息" → 存哪里？localStorage？数据库？session？

### 3. 安全相关
- 涉及密码、token、认证 → 必须确认实现方式
- 涉及用户数据 → 必须确认存储方式

### 4. 数据相关
- 涉及数据库 → 必须确认迁移策略
- 涉及字段类型变更 → 必须确认兼容性

### 5. 用户意图不明
- 任何不清楚"为什么"的情况

---

## 不追问的情况

仅当以下条件**全部满足**时，可以不追问：
- 复杂度为 trivial
- 意图明确无歧义
- 无安全/数据风险

此时输出简化版：

```
🤖 确认需求：

📋 [简短描述]
📁 涉及文件：[文件名]
⚡ 复杂度：trivial

直接执行？（y/n）
```

---

## 识别涉及文件的方法

通过项目结构分析：

```
执行命令：
ls -la                    # 查看项目根目录
find . -name "*.tsx"      # 查找组件文件
find . -name "*.ts"       # 查找 TypeScript 文件
cat package.json          # 查看项目配置
```

常见项目结构：
- `src/pages/` 或 `pages/` → 页面组件
- `src/components/` → 通用组件
- `src/services/` 或 `src/api/` → API 服务
- `src/store/` 或 `src/state/` → 状态管理
- `src/utils/` → 工具函数
- `src/db/` 或 `prisma/` → 数据库相关

## 示例

### 示例 1：trivial

**用户输入**：把登录按钮的颜色从蓝色改成绿色

**输出**：
```

📋 将登录按钮的背景色从蓝色改为绿色
📁 涉及文件：src/pages/Login.tsx
⚡ 复杂度：trivial

直接执行？（y/n）
```
