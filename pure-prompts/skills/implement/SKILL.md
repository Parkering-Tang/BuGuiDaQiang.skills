# implement - 安全实现

## 触发条件

在所有前置护栏（clarify、scope-guard、schema-guard 等）通过后触发。

---

## 核心约束

### 硬性规则

```
┌─────────────────────────────────────────────────────────┐
│ 🚨 绝对禁止行为                                           │
├─────────────────────────────────────────────────────────┤
│ ❌ 禁止修改 allowed_files 列表之外的文件                   │
│ ❌ 禁止"顺便"重构代码                                     │
│ ❌ 禁止添加未确认的功能                                    │
│ ❌ 禁止修改 API 契约（未经确认）                           │
│ ❌ 禁止修改数据库结构（未通过 schema-guard）               │
│ ❌ 禁止修改配置文件（未经确认）                            │
│ ❌ 禁止更改现有函数签名（未经确认）                        │
│ ❌ 禁止删除现有代码（除非是确认的目标）                    │
└─────────────────────────────────────────────────────────┘
```

---

## 执行前检查

在修改任何文件之前，**必须**执行：

```
□ 这个文件在允许列表中吗？
□ 这个修改是功能必需的吗？
□ 这个修改不是"顺便"做的吗？
□ 我已经读取文件内容了吗？
□ 我分析了文件的代码风格吗？
□ 新代码会遵循现有风格吗？
```

---

## 代码风格分析

**强制行为**：修改任何文件前，必须先读取并分析该文件。

### 分析清单

执行以下命令读取文件：
```bash
cat <文件路径>
```

然后分析：
```
📖 修改前风格分析

□ 缩进：空格/Tab？几个？
□ 引号：单引号/双引号？
□ 分号：使用/不使用？
□ 命名：camelCase/snake_case？
□ 组件风格：函数组件/类组件？
□ 状态管理：hooks/redux/其他？
□ 样式方式：CSS-in-JS/CSS Modules/inline？
□ 注释风格：JSDoc/普通注释/无注释？
□ 错误处理：try-catch/回调/其他？
□ 异步风格：async-await/Promise/回调？
```

### 风格一致原则

```
新增代码的风格 = 原有代码的风格

具体要求：
• 变量命名遵循现有模式
• 函数结构遵循现有模式
• 注释风格遵循现有模式
• 错误处理遵循现有模式
• 不引入新的风格模式
```

### 风格一致性示例

```typescript
// ❌ 错误：原文件用分号，新代码不用
const handleClick = () => {
  console.log('clicked')  // 原文件用分号，这里应该加;
}

// ✅ 正确：保持一致
const handleClick = () => {
  console.log('clicked');
};

// ❌ 错误：原文件用单引号，新代码用双引号
const name = "rememberMe"

// ✅ 正确：保持一致
const name = 'rememberMe';
```

## "顺便"行为判断

**判断标准**：

```
问自己：如果我不做这个修改，功能能正常工作吗？

✅ 能 → 这是"顺便"修改 → 禁止
❌ 不能 → 这是必要修改 → 允许
```

**禁止清单**：

| 禁止行为 | 原因 |
|---------|------|
| ❌ 顺便重构周围代码 | 超出范围 |
| ❌ 顺便统一命名风格 | 超出范围 |
| ❌ 顺便优化性能 | 超出范围 |
| ❌ 顺便修复其他 bug | 超出范围 |
| ❌ 顺便添加类型注解 | 超出范围 |
| ❌ 顺便改进错误处理 | 超出范围 |
| ❌ 顺便添加日志 | 超出范围 |

## 越界处理

如果发现需要修改范围外的文件：

```
⚠️ 需要超出原定范围

发现需要修改：src/store/user.ts
原因：登录状态持久化需要更新 store

原定范围：src/Login.tsx, src/services/auth.ts

是否允许扩展范围？
1. ✅ 允许，继续实现
2. ❌ 不允许，寻找替代方案
3. ⏸️ 暂停，重新评估
```
