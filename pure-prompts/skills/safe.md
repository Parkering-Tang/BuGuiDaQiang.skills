# safe

> 安全变更的主入口技能

## 触发方式

```
/safe [你的需求描述]
```

示例：
```
/safe 帮我改一下登录页面，加个记住密码功能
/safe 把用户状态改成存中文
/safe 优化一下首页加载速度
```

---

## 核心原则

在写任何代码之前，必须先完成：

1. **澄清需求** - 确保理解用户真正想要什么
2. **评估风险** - 判断改动的复杂度和潜在问题
3. **审查影响** - 分析会影响到哪些模块和数据
4. **创建快照** - 保存当前状态，以便回滚
5. **安全实现** - 在约束范围内执行修改
6. **验证完成** - 确保没有破坏现有功能

## 复杂度分级

| 等级 | 定义 | 典型场景 | 流程 |
|------|------|---------|------|
| **trivial** | 单文件、纯展示、无逻辑 | 改文案、调颜色 | 直接执行 → 验证 |
| **moderate** | 单模块、有逻辑但不跨边界 | 加表单字段 | 影响检查 → 快照 → 执行 → 验证 |
| **complex** | 跨模块、涉及 API/状态 | 新功能 | 完整护栏流程 |
| **high-risk** | 涉及数据库/认证/支付 | 改 schema | 强制多轮确认 + 详细审查 |

---

## 复杂度判断标准

### trivial（必须全部满足）
- [ ] 只涉及单个文件
- [ ] 不涉及业务逻辑
- [ ] 不涉及状态管理
- [ ] 不涉及 API 调用
- [ ] 不涉及数据库/存储

### moderate
- 涉及单个模块内的逻辑变更
- 可能涉及状态管理，但不跨模块
- 不涉及数据库 schema 变更

### complex
- 跨多个模块
- 涉及 API 契约变更
- 涉及共享状态变更

### high-risk（满足任一即触发）
- 涉及数据库 schema 变更
- 涉及认证/授权逻辑
- 涉及支付/资金相关
- 涉及敏感数据处理

## 快照创建命令

### 有 Git 时
```bash
# 记录当前 commit
git rev-parse HEAD

# 恢复命令
git restore .
git checkout <commit-hash>
```

### 无 Git 时
```bash
# 创建快照
mkdir -p .safe-vibecoding/snapshots/snap-YYYYMMDD-HHMM
cp -r <要备份的文件> .safe-vibecoding/snapshots/snap-YYYYMMDD-HHMM/

# 恢复快照
cp -r .safe-vibecoding/snapshots/snap-YYYYMMDD-HHMM/* .
```

## 验证命令

根据项目类型自动选择：

| 项目类型 | 构建命令 | 测试命令 |
|---------|---------|---------|
| Node.js | `npm run build` | `npm test` |
| Python | - | `pytest` |
| Rust | `cargo build` | `cargo test` |
| Go | `go build` | `go test ./...` |

检测方式：
- 存在 `package.json` → Node.js 项目
- 存在 `pyproject.toml` 或 `setup.py` → Python 项目
- 存在 `Cargo.toml` → Rust 项目
- 存在 `go.mod` → Go 项目

## 硬性规则

- ❌ 永远不要从模糊请求直接跳到代码
- ❌ 永远不要在没有数据审查的情况下修改 schema
- ❌ 永远不要在没有架构审查的情况下做跨模块变更
- ❌ 永远不要在没有验证的情况下声称完成
- ❌ 永远不要假设 Git 存在
- ❌ 永远不要假设用户理解数据模型变更的后果
